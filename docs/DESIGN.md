# MistakeTracker — Design

## 1. What we're building

A Flutter app that lets a user log everyday failures (missed deadlines, impulse spends, skipped workouts, arguments) and surfaces recurring causes, total cost, failure chains, and personalized prevention suggestions. v1 ships local-only with a manual JSON export. The architecture is set up so v2 can add cloud sync, LLM-powered suggestions, and context-aware notifications without rewriting.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Storage | Local SQLite via `sqflite` + JSON export | Zero backend risk, fastest to ship; export gives a backup story without auth |
| Intelligence | Rule-based `SuggestionEngine` v1, LLM-swappable | Reliable demo, explainable behavior, clean v2 extension |
| Notifications | Time-based only; `SignalSource` interface | Avoids permission/plugin risk; future signals (calendar/location/fitness) drop in |
| "My solution" field | In v1 | Tiny UX cost, big payoff: suggestion engine recommends user's own past fixes |
| Positive ("win") log | Deferred to v2 | Schema includes `kind` field so v2 only adds UI |
| State mgmt | Riverpod | Well-trodden pattern, good DI story for the swap interfaces |
| Platforms | Android first; iOS for free via Flutter | |

## 3. Architecture

```
lib/
├── app.dart                     # MaterialApp + bottom-nav routing
├── main.dart                    # (generated by `flutter create`)
├── data/
│   ├── database.dart            # sqflite open + migrations
│   ├── entry_dao.dart           # CRUD on entries
│   └── export_service.dart      # JSON writer + share intent
├── domain/
│   ├── models/
│   │   ├── entry.dart           # core data class (SHARED CONTRACT)
│   │   └── insight.dart         # output of suggestion engine (SHARED CONTRACT)
│   ├── suggestion_engine.dart   # abstract interface (SHARED CONTRACT)
│   └── rule_engine.dart         # v1 impl
├── notifications/
│   ├── signal_source.dart       # abstract interface (SHARED CONTRACT)
│   ├── time_signal.dart         # v1 impl
│   └── notifier.dart            # flutter_local_notifications glue
├── ui/
│   ├── home_screen.dart
│   ├── entry_form.dart
│   ├── insights_screen.dart
│   └── settings_screen.dart
└── state/
    └── providers.dart           # riverpod providers (DI + state)
```

**Dependency direction:** `ui` → `state` → (`domain`, `data`, `notifications`). No back-edges.

**Shared contracts** (changes need 3-person sign-off): the four files tagged `SHARED CONTRACT` above. They're the only thing two roles need to agree on; everything else each role can iterate on independently.

## 4. Data model

One table `entries` with columns matching `Entry.toMap()`:

```
id            INTEGER PRIMARY KEY AUTOINCREMENT
kind          TEXT NOT NULL DEFAULT 'failure'
what          TEXT NOT NULL
cause         TEXT
occurred_at   TEXT NOT NULL                   -- ISO-8601 UTC
context       TEXT
severity      INTEGER NOT NULL DEFAULT 3
cost_minutes  INTEGER
cost_money    INTEGER
mood_impact   TEXT
solution      TEXT
created_at    TEXT NOT NULL
```

Index on `occurred_at` for time-pattern queries.

## 5. v1 user flows

1. **Log a failure** — Home → "+" → form (auto-fills time) → save → home shows it on top.
2. **Browse history** — Home shows last N entries; tap to edit.
3. **See insights** — Insights tab. Engine runs over all entries, returns `Insight`s as cards.
4. **Get nudged** — When a time pattern is detected, schedule a local notification 1hr before the next at-risk window.
5. **Export** — Settings → "Export my data" → JSON file → Android share sheet.

## 6. Rule engine v1

Three rule families, all in plain Dart/SQL:

1. **Recurring cause** — group entries by `cause`; surface causes appearing ≥3 times in 30 days.
2. **Time-of-day pattern** — group by `(what, day_of_week)` and `(what, hour_of_day)`; flag combinations with ≥3 hits.
3. **Chain detection** — for each entry, look at the next entry within 6 hours; surface `(A → B)` pairs that appear ≥3 times.

Plus:
- **Suggestion text** = the user's own past `solution` for the same `what`, if any; else a generic templated tip.
- **Cost insight** = aggregate of `cost_minutes` and `cost_money`.

All thresholds live as constants in `rule_engine.dart` so they're tweakable in one place.

## 7. Notifications v1

- `flutter_local_notifications` for local scheduling. No push, no server.
- `TimeSignal` reads entries, detects time patterns, emits `NotificationTrigger`s as a stream.
- `Notifier` subscribes to any number of `SignalSource`s and schedules their triggers.
- v2 adds `CalendarSignal`, `LocationSignal`, `FitnessSignal` — no change to `Notifier`.

## 8. Error handling & edge cases

- **Empty state** — Insights/Home both show helpful empty-state copy instead of a blank screen.
- **DB failures** — DAO throws, UI catches with a SnackBar; never crash on a single insert.
- **Notification permission denied** (Android 13+) — graceful banner in Settings; app still works.
- **Export failure** — fallback: write to Downloads/ with a confirmation.
- **Timezone** — store UTC, display local.

## 9. Testing

Realistic for a 5-day team:

- **Unit tests** for `RuleEngine` — most important. Hand-craft 5–10 entry fixtures, assert expected insights.
- **Widget test** for `EntryForm` — round-trip: fill → save → appears in list.
- **No integration tests** in v1; manual checklist for the demo.

## 10. What's deliberately out of v1

These are *designed* but not built. Each has a documented extension point so v2 is additive, not a rewrite.

| Feature | Extension point |
|---|---|
| Cloud sync (Firebase) | `EntryDao` interface — swap impl |
| LLM-powered suggestions | `SuggestionEngine` interface — add `LLMEngine` impl |
| Calendar-aware notifications | `SignalSource` interface — add `CalendarSignal` impl |
| Location-aware notifications | `SignalSource` interface — add `LocationSignal` impl |
| Fitness-aware notifications | `SignalSource` interface — add `FitnessSignal` impl |
| Positive ("win") log | `Entry.kind` field already supports it; add a second log UI |
| Home-screen widgets | New Android-side AppWidgetProvider |
